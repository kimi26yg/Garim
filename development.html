📡 실시간 영상 통신 구조 (WebRTC & Signaling)

garim_eye_v2는 카카오 페이스톡, 인스타그램 영상 통화와 같은 실제 화상 통화 서비스의 인프라를 그대로 재현하였습니다.

1. 시그널링 서버 (Node.js + Socket.IO)

  • 목적: 기기 간 초기 연결을 중계하는 서버로 현대 화상 통화 서비스 인프라 재현
  • 번호 기반 라우팅: 유동적인 소켓 ID 대신 고유한 010 가상 번호를 식별자로 사용
  • 이중 매핑: phoneRegistry를 통해 번호와 소켓 ID를 매핑 관리
  • 배포: Railway 클라우드 플랫폼에서 상시 가동 (garim_attack_server)
  • 신호 교정: 웹/앱 환경 차이로 발생하는 데이터 규격 차이를 서버에서 교정

2. WebRTC P2P 미디어 스트리밍

  • 초저지연 전송: 서버를 거치지 않고 사용자 기기 간 직접 교환
  • Remote Stream 접근: 상대방의 RemoteVideoTrack에서 프레임 데이터를 직접 가로채 분석
  • 보안: DTLS/SRTP 암호화 전송, 영상 데이터는 서버에 미저장
  • 구조: 웹(공격자) ↔ 서버(중계기) ↔ 앱(방어자 + G-EYE)

3. 통합 통신 규격 (Protocol v1.1 Unified)

웹 대시보드와 모바일 앱이 오차 없이 대화할 수 있도록 독자적인 통신 표준 구축:

  • register:phone - 사용자 가상 번호 등록
  • call:request - 보안 통화 세션 시작 요청
  • call:response - 수락 시 WebRTC 핸드셰이킹 개시
  • webrtc:offer/answer - 영상 포맷 및 코덱 정보 교환
  • call:hangup - 통화 종료 및 자원 반납

🧠 온디바이스 AI 모델 이식 과정

실시간 딥페이크 탐지를 위해 MobileNetV3 + LSTM 모델을 Flutter 앱에 직접 이식하는 과정에서 여러 기술적 도전 과제를 해결했습니다.

⚠️ 이슈 1: TFLite 모델 추론 점수가 일관되게 0.0 출력

증상:
  • Python 환경에서는 정상 작동하는 모델이 Flutter TFLite에서 항상 0.0 반환
  • BATCH_SIZE 및 입력 버퍼 로직 수정 후에도 동일한 결과
  • 로그 출력이 누락되어 디버깅 어려움
원인 분석:
  • 모델 변환 과정에서의 구조적 문제 의심
  • 입력 텐서 형식 불일치 (shape, dtype)
  • 로깅 시스템이 제거되어 중간 추론 과정 불투명
해결 방법:
  • Python에서 모델 테스트를 통한 변환 전후 일관성 검증
  • 입력 전처리 파이프라인 재검증 (정규화, 리사이징)
  • 상세 로깅 재구현으로 추론 각 단계별 모니터링 강화
  • TFLite Interpreter 초기화 시 delegate 옵션 최적화

🔴 이슈 2: iOS Xcode 빌드 실패 (PhaseScriptExecution)

증상:
  • Command PhaseScriptExecution failed 에러 발생
  • xcode_backend.dart에서 Null check operator 오류
  • Flutter xcconfig 파일 참조 실패
해결 방법:
  • project.pbxproj 파일 내 Flutter xcconfig 경로 수정
  • FLUTTER_BUILD_DIR 환경 변수 인식 문제 해결
  • Podfile 및 빌드 설정 재검증
  • Clean build 수행으로 캐시 문제 제거

📦 이슈 3: Socket.IO 데이터 파싱 오류

증상:
  • 웹에서 전송된 데이터가 앱에서 List 형태로 래핑되어 수신
  • SDP payload가 String으로 전달되는 경우 파싱 실패
  • 런타임 타입 에러 발생
해결 방법:
  • 데이터 수신 시 List 여부 먼저 감지
  • List인 경우 첫 번째 요소 추출 후 Map.from()으로 안전 변환
  • String SDP는 JSON 파싱 후 처리
  • 서버 측 신호 교정 로직 추가로 근본 해결

🌐 이슈 4: 웹 환경 얼굴 탐지 (MediaPipe 마이그레이션)

배경:
  • ML Kit은 Flutter Web에서 미지원
  • 웹 환경에서도 얼굴 탐지 기능 필요 (garim_attack 웹 대시보드)
해결 방법:
  • google_mlkit_face_detection → mediapipe_tasks_vision 교체
  • MediaPipe WASM 백엔드를 이용한 웹 전용 얼굴 탐지 구현
  • CanvasElement, ImageElement 기반 프레임 처리로 변경
  • 크로스 플랫폼 일관성 유지

💡 개발 인사이트

  • 실제 서비스 환경 재현: 일반 화상 통화 앱과 동일한 WebRTC 구조를 채택하여 B2B 파트너사가 기존 인프라에 쉽게 통합할 수 있도록 설계했습니다.
  • 온디바이스 AI의 현실: 서버 의존 없이 사용자 기기에서 직접 추론을 수행하면서 모델 최적화, 플랫폼별 호환성, 성능 이슈 등 다양한 도전 과제를 경험했습니다.
  • 디버깅의 중요성: 상세한 로깅과 단계별 검증이 없다면 블랙박스 같은 모바일 AI 모델 디버깅은 사실상 불가능합니다. 투명성 확보가 핵심입니다.
  • 크로스 플랫폼 통합: 웹(공격자 시뮬레이션)과 앱(실제 사용자)이 완벽히 통신하려면 프로토콜 표준화와 데이터 교정 로직이 필수입니다.

✨ V5.1 UI/UX 고도화 기술 명세

사용자 경험을 극대화하기 위한 미세 조정 (Micro-interactions)

1. 4-Way Ambient Glow (물리 엔진 기반)

위험도(RiskLevel)에 따라 펄스 애니메이션의 물리적 속성이 실시간으로 변화합니다.

상태 애니메이션 컨트롤러 Glow 두께 (Thickness) 투명도 (Opacity)
Safe Static 40.0px (고정) 15%
Warning Slow Pulse (1000ms) 70px ↔ 85px 25% ↔ 40%
Critical Rapid Pulse (500ms) 100px ↔ 130px 30% ↔ 80%

2. 다이내믹 햅틱 피드백 (Haptic)

  • Warning: MediumImpact (1회) - 사용기 주의 환기.
  • Critical: HeavyImpact (즉시) + 1초 주기 반복 - 확실한 경고 전달.
  • 상태가 Safe로 복귀 시 타이머 즉시 해제 (Debouncing 적용).

3. 지능형 배터리 세이버 (Auto-Stop)

  • Trigger: 통화 연결 후 5분 이상 DetectionStatus.safe 지속 시 발동.
  • Action: AI 엔진을 Paused 상태로 전환하여 NPU/GPU 자원 점유율 0%로 하락.
  • UX: "Safe Call Confirmed" 다이얼로그 출력 후 보호 모드 토글 자동 OFF.

4. Security Insight Panel 상세

개발자 및 QA를 위한 실시간 블랙박스 데이터 시각화 도구입니다.

  • 등급 세분화: 95점 이상 "ULTRA HD SAFE", 80점 이상 "SAFE", 40점 미만 "DANGER".
  • 가변 샘플링: 성능과 발열 제어를 위해 1.25s, 5.0s, 10.0s 주기를 동적으로 스위칭.
  • 열 관리 (Thermal): 기기 온도 상태(Nominal, Fair, Critical)에 따라 UI 색상 변경 (Green -> Red).
